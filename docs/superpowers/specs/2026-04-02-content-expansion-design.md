# 内容深度扩展设计文档

## 项目背景

测试开发面试速成站（TestDev-Sprint）当前内容条目数量充足，但各章节内容深度不足。用户反馈："各个章节的内容都有点少，应该面向初学者级别的用户。"

本设计文档规划如何扩展 Tech、Scenario、Coding 三大分类的内容深度，使其形成小教程级别的篇幅，帮助初学者解决概念不清、落地难、面试表达困难、学习路径迷茫四大痛点。

## 需求总结

### 核心需求
- **目标分类**：Tech、Scenario、Coding（优先扩展）
- **目标用户**：初学者级别用户
- **篇幅预期**：小教程级别（每个条目形成完整知识块）
- **核心痛点**：
  - 概念不清：不知道某个技术/概念是什么
  - 不知如何落地：知道概念但不知道项目里怎么用
  - 面试表达困难：有经验但面试时讲不清楚
  - 学习路径迷茫：不知道从哪里开始学

### 特别关注的技术主题
- Python/Pytest
- 接口测试
- Web 自动化（Playwright）
- 性能测试

## 方案选择

**选择方案 A：渐进式扩展**

每个现有条目按统一模板扩展，保持现有结构不变。

**优势**：
- 改动可控，降低开发风险
- 可分批次扩展，每批验证效果
- 用户导航路径保持，不影响现有用户
- 条目之间独立，支持并行执行

## 内容模板设计

### Tech 分类扩展模板

每个 Tech 条目扩展为以下 5 个模块：

#### 模块 1：基础入门（合并"是什么"和"为什么学"）

**内容要求**：
- 是什么：详细解释概念、原理、核心能力
- 为什么学：说明学习价值、面试重要性、与项目的关系
- 与竞品对比：如有竞品（如 Pytest vs unittest），说明差异
- 前置知识：列出必须掌握和建议掌握的知识

**示例结构**：
```
## 基础入门：Pytest 是什么 & 为什么学它

### Pytest 是什么
（300-500 中文字详细解释，包含核心能力列表）

### 为什么测试开发要学 Pytest
（5个理由，每个100-200 中文字）

### Pytest vs unittest：为什么 Pytest 更适合新项目
（对比表格）

### 前置知识：学 Pytest 前你需要会什么
（必须掌握、建议掌握、不需要掌握）
```

#### 模块 2：学习路径

**内容要求**：
- 零基础第一步：10分钟快速体验任务
- 分阶段学习：4个阶段，每个有目标、内容、练习任务
- 时间建议：不同时间投入的学习周期
- 学习资源：推荐文档、书籍、实践建议

**示例结构**：
```
## 学习路径：从零到能写项目级测试

### 零基础第一步：如果你完全没接触过 Pytest
（10分钟快速体验任务）

### 分阶段学习建议
- 第一阶段：基础语法（目标、内容、练习任务）
- 第二阶段：Fixture 和资源管理
- 第三阶段：参数化与数据驱动
- 第四阶段：工程化能力

### 时间投入建议
（每天1-2小时和每天30分钟的周期）

### 学习资源推荐
（官方文档、书籍、实践建议）
```

#### 模块 3：实操案例

**内容要求**：
- 入门案例（案例0）：写第一个测试，10分钟完成
- 基础案例（案例1）：核心结构，理解整体架构
- 进阶案例（案例2-3）：复杂场景，面试加分项
- 所有案例用流程描述而非完整代码
- 每个案例有难度标注和关键设计要点解读

**示例结构**：
```
## 实操案例：从流程理解真实应用

### 案例难度说明
（表格标注入门/基础/进阶）

### 案例 0：写你的第一个 Pytest 测试（入门）
（步骤描述、学到什么、下一步）

### 案例 1：搭建一个接口自动化项目骨架（基础）
（架构流程图、核心流程描述、关键要点表格）

### 案例 2：测试异步回调接口（进阶）
（流程图、测试流程、关键要点）

### 案例 3：使用 Mock 测试依赖隔离（进阶）
（Mock 流程图、使用流程、关键要点）
```

#### 模块 4：常见误区

**内容要求**：
- 5个常见误区
- 每个误区包含：错误表现、为什么错、正确做法、面试应对
- 用流程描述而非代码示例
- 最后有误区总结对照表

**示例结构**：
```
## 常见误区：初学者最容易踩的坑

### 误区 1：把所有 Fixture 都写在根目录 conftest.py
（错误表现表格、正确做法流程、面试回答逻辑）

### 误区 2：Fixture 作用域选择一刀切
（判断流程、选择表格、面试回答）

### 误区 3-5：（同理）

### 误区总结
（对照表）
```

#### 模块 5：面试问答

**内容要求**：
- 5个高频问题
- 每个问题包含：回答骨架、深度答案、追问应对
- 标注问题优先级（P0/P1）
- 最后有面试回答核心技巧表

**示例结构**：
```
## 面试问答：如何把知识讲清楚

### 问题优先级说明
（表格标注频率、难度、优先级）

### Q1：你如何设计 Pytest 项目结构？
（回答骨架、深度答案、追问应对表格）

### Q2-5：（同理）

### 面试回答的核心技巧
（技巧表格）
```

### Scenario 分类扩展模板

每个 Scenario 条目扩展为以下 4 个模块：

#### 模块 1：场景背景
- 业务背景：这个场景出现在什么项目里
- 为什么重要：面试高频原因
- 常见项目类型：哪些项目会涉及

#### 模块 2：回答框架
- 回答结构：标准答题骨架
- 关键要点：必须覆盖的内容点
- 时间控制：回答时长建议

#### 模块 3：追问应对
- 3-5个常见追问
- 每个追问的应对策略和示例回答

#### 模块 4：实战案例
- 2-3个不同项目类型的实际场景
- 每个场景的回答示例

### Coding 分类扩展模板

每个 Coding 条目扩展为以下 5 个模块：

#### 模块 1：题目背景
- 为什么问这个：考察什么能力
- 题目类型：设计类、实现类、优化类

#### 模块 2：解题思路
- 思考步骤流程
- 设计选择和权衡考虑

#### 模块 3：代码逻辑（流程描述）
- 核心逻辑流程图
- 关键步骤描述
- 不展示完整代码，用流程描述

#### 模块 4：常见失分点
- 3-5个失分点
- 每个包含：错误做法、为什么不好、如何改进

#### 模块 5：进阶讨论
- 性能考虑
- 扩展性
- 其他方案对比

## 实施优先级

### 第一批扩展（高优先级）

**Tech 分类（4个条目）**：
| 条目 slug | 条目标题 | 原因 |
|-----------|---------|------|
| python | Python 测试开发基础 | 基础必学，面试高频 |
| pytest | Pytest 测试框架 | 基础必学，面试高频 |
| api-testing | 接口测试 | 用户特别关注 |
| playwright | Playwright Web 自动化 | 用户特别关注 |
| performance-testing | 性能测试 | 用户特别关注 |

**Scenario 分类（2个条目）**：
| 条目 slug | 条目标题 | 原因 |
|-----------|---------|------|
| payment-callback | 支付回调场景题 | 高频场景题 |
| login-auth | 登录鉴权场景题 | 高频场景题 |

**Coding 分类（3个条目）**：
| 条目 slug | 条目标题 | 原因 |
|-----------|---------|------|
| retry-mechanism | 请求重试编码题 | 高频编码题 |
| assertion-wrapper | 断言封装编码题 | 高频编码题 |
| fixture-strategy | Fixture 设计编码题 | 高频编码题 |

**第一批条目总数**：9 个（Tech 5个 + Scenario 2个 + Coding 3个，其中性能测试也属于 Tech 分类，共 5 个 Tech 条目）

### 第二批扩展（中优先级）

| 分类 | 条目 | 原因 |
|------|------|------|
| Tech | CI/CD、Mock、数据驱动 | 面试常问但相对独立 |
| Scenario | 平台建设、秒杀场景 | 进阶场景题 |
| Coding | 日志封装、数据驱动 | 补充编码能力 |

### 第三批扩展（低优先级）

| 分类 | 条目 | 原因 |
|------|------|------|
| Tech | SQL、Linux | 基础但非核心 |
| Scenario | 兼容性测试、推荐系统 | 相对低频 |

## 并行执行设计

由于条目之间独立，本设计支持多 subagent 并行执行：

### 并行策略 1：按分类并行
- Agent A：扩展 Tech 分类第一批条目
- Agent B：扩展 Scenario 分类第一批条目
- Agent C：扩展 Coding 分类第一批条目

### 并行策略 2：按条目并行
- 每个 subagent 负责 1-2 个条目
- 所有 subagent 使用统一的扩展模板
- 完成后统一 review 质量

### 并行策略 3：按模块并行（不推荐）
- 同一条目的不同模块由不同 subagent 完成
- 需要额外协调，增加复杂度
- 不建议采用

**推荐**：策略 2（按条目并行），每个 subagent 独立完成一个条目的全部模块。

## 质量标准

### 内容深度标准
- 基础入门模块：300-500 中文字核心解释 + 对比表格 + 前置知识
- 学习路径模块：4阶段 + 每阶段目标/内容/练习
- 实操案例模块：流程图 + 流程描述 + 关键要点表格
- 常见误区模块：5个误区 + 每个误区4部分内容
- 面试问答模块：5个问题 + 回答骨架/深度答案/追问

### 初学者适配标准
- 所有代码逻辑用流程描述而非完整代码
- 术语首次出现时有解释或说明
- 有"零基础第一步"快速体验任务
- 有难度标注和优先级标注
- 有面试回答骨架帮助表达

### 验收标准
- 每个条目总篇幅：1500-2500 中文字（含标点符号）
- 流程描述清晰，无需看代码即可理解逻辑
- 面试回答可以直接用于面试准备
- 初学者看完能知道"是什么、怎么学、怎么用、怎么答"

## 风险与应对

| 风险 | 应对措施 |
|------|---------|
| 信息过载 | 分批次扩展，第一批先完成高频条目 |
| 内容质量不一致 | 统一模板 + 每个 subagent 完成后 review |
| 术语解释缺失 | 模板要求术语首次出现时解释 |
| 面试问答不够实战 | 每个问题必须有追问应对和回答示例 |

## 后续步骤

1. 用户审核设计文档
2. 调用 writing-plans skill 创建实施计划
3. 按计划分批次实施扩展
4. 每批次完成后 review 质量
5. 根据反馈调整后续批次
