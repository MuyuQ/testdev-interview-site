# 文档站纸质主题设计

## 背景

当前站点已经开始往“更适合阅读的内容站”方向收敛，但视觉骨架仍然带有较强的默认文档站 / 卡片站痕迹：

- 亮色模式偏冷白、偏组件化，阅读氛围不足。
- 暗色模式虽然可用，但更像深色 UI，而不是夜间阅读用的深色文档页。
- 多处使用圆角卡片与面板，整体更接近产品卡片站，而不是成熟的文档系统。

本次设计目标不是简单换一组颜色，而是将全站亮暗主题统一重构为一套“纸页阅读”体系，并让整体结构向 [Zensical Docs](https://zensical.org/docs/get-started/) 这类更克制的文档站学习：清晰的栏位、稳定的边线、无圆角的文档骨架、强排版感，而不是大量悬浮卡片。

## 目标

- 将亮色模式升级为“暖米白阅读纸”主题。
- 将暗色模式升级为“深墨纸页”主题。
- 让亮暗两套主题成为同一套阅读设计语言，而不是简单的明暗翻转。
- 全站移除圆角边，统一为直角、细边框、文档壳层级。
- 让首页、内容页、导航、TOC、表格、代码块、提示块和交互控件都遵循同一套设计规则。
- 整体结构和气质向成熟文档站靠拢，减少“功能卡片站”观感。

## 非目标

- 不做复古旧纸、做旧书页、噪点纸纹等强风格化效果。
- 不复制参考站的品牌视觉，只借鉴其文档骨架、节奏与克制感。
- 不重做信息架构，不改栏目划分与内容组织方式。
- 不为了“纸感”牺牲可读性、对比度或无障碍基础。

## 用户体验目标

- 用户首次进入站点时，应感受到这是一个“为阅读和理解而设计”的文档站，而不是一组功能卡片拼贴。
- 亮色模式应有温和、柔和、长时间阅读友好的纸面气质。
- 暗色模式应有沉静、深墨、夜读友好的文档气质，而不是发亮的深色面板感。
- 首页与正文页之间应保持同一套语言：相同的壳层、边线、标题秩序和内容密度。
- 主题切换后，用户应感到是“白天纸页 / 夜间墨页”的切换，而不是两套毫无关系的皮肤。

## 最终设计方向

### 亮色模式

最终选定方向为：`暖米白阅读纸`

核心特征：

- 主背景为浅暖米白，接近高质量阅读纸而不是纯白网页。
- 面板、正文区、侧栏与工具区通过细微明度差形成纸页叠层。
- 文本使用暖深灰和暖灰褐，而不是纯黑。
- 强调色使用低饱和墨蓝，像墨水标记而不是科技蓝。

### 暗色模式

最终选定方向为：`深墨纸页`

核心特征：

- 主背景为深墨黑，带轻微蓝灰感，像夜间阅读纸页而非纯黑屏幕。
- 内容区和导航区以较浅一层的墨灰纸面承载信息。
- 文本采用柔和浅灰，不使用刺眼白字。
- 强调色延续墨蓝体系，在暗色下更像蓝墨批注。

### 共同骨架

两套主题共享以下规则：

- 整体向 Zensical 一类文档站的壳层与排版节奏学习。
- 去除圆角边，改用直角和细边框。
- 以边线、分栏、留白和排版层级建立秩序，而不是依赖厚重阴影和卡片漂浮。
- 首页也纳入文档壳，而不是独立的营销卡片首页。

## 参考站借鉴原则

参考目标：[Zensical Docs / Get Started](https://zensical.org/docs/get-started/)

本次借鉴的是以下结构语言，而不是视觉复制：

- 清晰稳定的文档壳：左侧导航、中间正文、右侧目录
- 面板和正文以直角边界分隔
- 更依赖 typography、spacing、rules（分隔线）建立层级
- 组件感弱，文档感强
- 页面中几乎没有“圆润卡片集合”的产品页观感

落到本项目中，意味着：

- 首页需要保留内容入口功能，但视觉上更像文档目录页，而不是独立营销页。
- 所有内容模块都要减少“卡片块状感”，增强“文档栏目感”。
- 边框、分栏、表格、代码块、引用块都要更统一，像同一份讲义系统。

## 作用范围

本次设计覆盖全站亮暗主题，包括：

- 首页
- 内容详情页
- 左侧导航
- 顶部栏 / 工具区
- 右侧 TOC
- 搜索与筛选控件
- 卡片类入口模块
- 表格、代码块、引用、提示块
- 分享按钮、分页、最近浏览等局部交互区

## 视觉规则

### 1. 形状系统

- 全站取消圆角边。
- 面板、卡片、按钮、输入框、表格、代码块、提示块、标签均使用直角。
- 形状语言从“圆润产品卡片”切换为“文档盒体 / 纸页框线”。

### 2. 页面骨架

- 重点强化文档站三栏阅读结构：导航、正文、目录。
- 首页也采用更接近正文页的排版秩序，而不是孤立的视觉系统。
- 不再依靠大面积漂浮卡片制造层级，改用分栏、边线与留白控制节奏。

### 3. 亮色背景层级

- 页面主背景使用浅暖米白。
- 正文区、侧栏、目录区和局部模块采用略有区分的浅暖层级。
- 层级差应轻微但稳定，让用户感受到纸页与内页之间的分化。

### 4. 暗色背景层级

- 页面主背景使用深墨黑与蓝灰混合的深纸面颜色。
- 正文区与工具区比背景稍亮一层，像深纸上的内页。
- 所有深色层级都应克制，避免“亮块贴在黑底上”的后台感。

### 5. 文本体系

- 亮色标题使用暖深灰褐，正文使用暖深灰，辅助文字更浅一档。
- 暗色标题使用柔和近白灰，正文使用偏灰浅色，辅助文字略沉。
- 两个主题都避免极端对比：亮色不用纯黑，暗色不用纯白。
- 通过字体大小、粗细、字距和块间距建立信息层级，而不是只靠颜色。

### 6. 强调色体系

- 链接、当前项、按钮重点态统一为低饱和墨蓝。
- 亮色中像纸面墨迹，暗色中像蓝墨批注。
- focus、selected、hover 保留明确反馈，但避免高亮发光或高饱和科技感。

### 7. 边框与分隔

- 使用细、稳定、低存在感的边框。
- 亮色使用暖灰线，暗色使用偏中性的灰蓝线。
- 边框负责建立壳层和阅读秩序，应比阴影承担更多结构责任。
- `hr`、表格线、侧栏分隔、代码块边界、提示框边界都需统一。

### 8. 阴影与层次

- 阴影仅作为极轻的辅助，不再作为主要层次手段。
- 亮色可以保留非常轻的暖阴影，暗色进一步减弱。
- hover 态以边框、背景和细微位移为主，不做“悬浮卡片”效果。

### 9. 代码块与结构化内容

- 亮色代码块使用浅米灰或浅纸灰背景，像讲义摘录区。
- 暗色代码块使用更深一层的墨页背景，与正文区区分开。
- 表格、引用、提示块要更像文档中的批注与嵌入段落，而不是组件面板。
- 代码块、表格、引用块、提示块全部使用直角与统一边框语言。

### 10. 首页模块

- 首页 hero、路径卡、模块入口、最近浏览、自测区等需要保留信息效率，但不再表现为一组圆角白卡片。
- 首页模块应该更像“文档目录页中的内容分栏与章节入口”。
- 鼠标悬停反馈保持克制，主要依赖边线、底色与轻微位移变化。

## 实现边界与结构建议

### 主题变量

以 `src/styles/tokens.css` 作为亮暗主题的统一入口：

- 亮色变量整体改为暖米白纸页体系
- 暗色变量整体改为深墨纸页体系
- 背景、文字、边框、强调色、结构化内容背景应在 tokens 中先统一定义

### 全局布局

以 `src/styles/custom-layout.css` 调整全站文档壳层与排版：

- 页面背景与正文承载区
- 侧栏、顶部栏、右侧 TOC
- 标题、正文、列表、引用、表格、分隔线
- 首页与正文页共享的文档壳节奏

### 组件层

以 `src/styles/components.css` 调整交互与局部模块：

- 按钮、输入框、标签、筛选项
- 分享按钮
- 搜索框与分页
- 最近浏览、自测题、首页模块入口

### 首页专项

首页可以保留内容聚合和快速入口职责，但需要：

- 跟随统一 tokens 和 layout 变量
- 从“卡片首页”收敛为“文档目录首页”
- 与正文页共享边框、层级、标题和控件语言

## 测试与验证

### 自动化验证

- 扩展现有主题 e2e，用例覆盖亮暗主题下的关键页面。
- 重点断言首页与内容页在亮暗模式下的 `data-theme`、关键背景层级、边框或正文区域状态。
- 确认 `npm run check` 与 `npm run test:e2e` 持续通过。

### 人工验证

至少人工检查以下场景：

- 首页亮色模式
- 首页暗色模式
- 内容页亮色模式
- 内容页暗色模式
- 左侧导航、右侧 TOC、顶部栏
- 代码块、表格、引用、提示块
- 搜索框、分页、分享按钮、hover/focus
- 桌面端与移动端

## 风险与控制

### 风险 1：纸感过强，变成复古风或装饰风

控制方式：

- 不使用纸纹图片、噪点贴图、旧纸黄调
- 通过色温、边框、排版和层级表达材料感

### 风险 2：去圆角后显得过硬或过旧

控制方式：

- 依赖更好的 spacing、边框节奏和文字层级来补足精致度
- 保持边框细而稳，不用厚描边

### 风险 3：参考站借鉴过度，失去自身内容站特征

控制方式：

- 只借鉴结构语言，不复制品牌细节
- 保留本站中文内容密度、首页入口组织和学习路径特征

### 风险 4：暗色模式阅读舒适度下降

控制方式：

- 避免纯白文字和高饱和强调色
- 保持正文、背景、结构区之间稳定但温和的对比

## 交付标准

当以下条件满足时，本次设计可认为完成：

- 亮色模式完整呈现“暖米白阅读纸”气质
- 暗色模式完整呈现“深墨纸页”气质
- 两套主题在结构、强调色、边框和排版上属于同一套设计语言
- 全站已移除圆角边，统一为直角文档盒体
- 首页、内容页、导航与结构化内容都已完成统一收敛
- 整体观感更接近成熟文档站，而不是卡片式功能首页
- 自动化检查与浏览器验证通过
