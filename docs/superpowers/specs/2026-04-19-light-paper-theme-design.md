# 亮色模式纸质感主题设计

## 背景

当前站点的亮色模式整体偏“默认文档站”风格：白底、冷灰边框、偏冷的强调色。这种方案虽然清晰，但与站点当前内容气质不完全匹配，也缺少更稳定的阅读氛围。

本次设计希望将亮色模式统一调整为“轻纸感”方向，让页面更像高质量阅读纸，而不是纯白后台。目标不是复古化，也不是做旧，而是在不破坏信息效率和可读性的前提下，让亮色模式更温和、更有材料感。

## 目标

- 将全站亮色模式统一升级为“暖米白阅读纸”风格。
- 保持现代文档站的清晰度、可读性与专业感。
- 让首页、内容页、导航区、卡片区和交互控件在同一套亮色主题语言下工作。
- 暗色模式保持现状，不引入不必要的变化。

## 非目标

- 不做复古旧纸、发黄纸页、做旧书页效果。
- 不引入明显纸张纹理图片、噪点背景或强装饰图案。
- 不重做整体信息架构。
- 不改变暗色模式的视觉方向。

## 用户体验目标

- 首次切换到亮色模式时，用户应感受到更柔和、更适合长时间阅读的界面氛围。
- 亮色模式中的层级关系应该更像纸页叠放，而不是大量白色方块拼接。
- 链接、选中态和重点信息应有更稳定的“墨水感”，而不是偏冷的科技蓝。
- 在首页与内容页之间切换时，视觉语言应保持连续，不出现“首页一套、正文一套”的割裂。

## 设计方向

最终选定方向为：`A. 暖米白阅读纸`

这一路线的核心特征：

- 主背景为浅暖米白，接近未漂白的高质量阅读纸。
- 卡片、面板与导航区使用更亮或更浅的暖色层级，形成纸页叠层感。
- 正文字体颜色从冷黑调整为暖深灰，降低刺眼感。
- 强调色从当前偏冷蓝调整为低饱和墨蓝，保留专业感与阅读秩序。
- 纸感主要通过色温、层级、细边框和轻阴影体现，而不是依赖强纹理。

## 作用范围

本次设计覆盖亮色模式的全站表现，包括：

- 首页
- 内容详情页
- 侧边栏导航
- 顶部导航与工具区
- 卡片类组件
- 表格、引用、代码块、提示块
- 搜索框、筛选按钮、分页、分享按钮等常见控件

## 视觉规则

### 1. 背景层级

- 页面主背景使用浅暖米白，而不是纯白。
- 内容面板、首页卡片、导航面板使用略亮的纸页层级，与主背景形成轻微分层。
- 层级差应该足够可感知，但不应明显到变成“彩色模块拼贴”。

### 2. 文本体系

- 主标题使用偏深的暖灰褐色，保证权重与清晰度。
- 正文使用暖深灰，保持长文阅读舒适度。
- 次级说明文字使用更浅一档的暖灰，避免冷灰带来的生硬感。
- 整体避免纯黑，以降低亮色模式下的视觉刺激。

### 3. 强调色

- 链接、当前项、按钮重点态统一收敛到低饱和墨蓝。
- 强调色应更像“墨水标记”而不是荧光式高亮。
- hover、focus、selected 的变化保留，但避免高亮过强。

### 4. 边框与分隔

- 使用细、浅、暖灰系边框和分隔线。
- 目标是“纸面压线感”而不是冰冷的 UI 描边。
- 在卡片、代码块、表格、引用块之间形成稳定的文档秩序。

### 5. 阴影

- 卡片与悬浮控件仅保留极轻的暖色阴影。
- hover 允许非常克制的抬升感。
- 不做厚重浮层或明显投影。

### 6. 代码块与结构化内容

- 亮色模式下的代码块背景改为浅米灰，而不是冷调浅灰。
- 表格表头与引用块背景也统一采用纸面注释式层级。
- 代码、表格、引用应看起来像讲义中的嵌入内容，而不是后台管理组件。

### 7. 首页卡片

- 首页的路线卡、模块卡、进度区、最近浏览区要与全站纸质亮色保持一致。
- 卡片应像纸页轻叠，而不是默认白卡片。
- 鼠标悬停时只做轻微边框与阴影变化，避免“科技卡片”的悬浮感。

### 8. 纹理与装饰

- 不引入明显纸张纹理图片。
- 允许非常轻的色差、暖渐变或局部层次来暗示材料感。
- 一切装饰都必须让位于阅读，不得抢正文注意力。

## 实现方案

### 主题变量

以 `src/styles/tokens.css` 作为亮色模式的主题基线来源：

- 重定义亮色模式下的背景变量
- 重定义亮色模式下的文字层级变量
- 重定义亮色模式下的边框变量
- 重定义亮色模式下的强调色变量
- 保持暗色模式变量不变

### 全局布局

以 `src/styles/custom-layout.css` 负责全站文档层级表现：

- 标题颜色
- 正文与列表颜色
- 表格、blockquote、hr
- 内容面板背景与层级
- 侧边栏与当前导航项亮色表现

### 组件层

以 `src/styles/components.css` 负责交互控件和卡片类组件：

- 卡片背景与边框
- 按钮与筛选项
- 搜索框
- 分享按钮
- 分页链接
- 最近浏览与其他局部组件

### 首页专项

首页仍保留局部样式，但需确保：

- 优先复用全局变量
- 不再使用“纯白卡片”作为亮色默认值
- hero、路线卡、模块卡、进度区与最近浏览区跟随纸质亮色体系

## 测试与验证

### 自动化验证

- 保留并扩展现有 e2e 主题测试。
- 新增或调整亮色主题断言，覆盖首页和内容页中的关键背景色、文字色或主题属性。
- 确认 `npm run check` 与 `npm run test:e2e` 继续通过。

### 人工验证

需要至少人工检查以下场景：

- 首页亮色模式
- 内容页亮色模式
- 侧边栏与当前项状态
- 代码块、表格、引用
- 搜索框、分页、按钮 hover/focus
- 桌面端与移动端

## 风险与控制

### 风险 1：纸感过重导致“复古化”

控制方式：

- 不使用明显旧纸颜色
- 不加入强纹理或做旧效果
- 强调保持现代文档站秩序

### 风险 2：亮色可读性下降

控制方式：

- 保持正文与背景之间足够对比
- 不把正文变成浅棕或低对比颜色
- 优先以阅读舒适度而不是风格感为第一准则

### 风险 3：首页与内容页割裂

控制方式：

- 首页局部样式全面切到共享变量体系
- 通过主题变量统一控制颜色层级

## 交付标准

当以下条件满足时，本次设计可认为完成：

- 亮色模式整体呈现出稳定的暖米白阅读纸质感
- 首页与内容页视觉语言统一
- 强调色、边框、卡片、正文、代码块与表格均已完成亮色主题收敛
- 暗色模式未被破坏
- 自动化检查与浏览器测试通过
