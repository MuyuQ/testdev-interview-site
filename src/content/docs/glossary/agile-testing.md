敏捷测试是在短周期迭代中持续参与质量保障的方式。它不是“测试时间被压缩”，而是要求测试更早介入、更快反馈、更重视协作和自动化。测试人员需要从需求评审开始识别风险，而不是等开发完成后才找问题。

敏捷测试常见实践包括测试左移、三方需求澄清、验收标准定义、持续集成、自动化回归、探索式测试、快速缺陷反馈和迭代复盘。测试开发在敏捷团队里通常还要维护自动化框架、Mock 能力、测试数据工具和质量看板，帮助团队缩短反馈周期。

面试回答敏捷测试时，可以按“提前参与、持续验证、快速反馈、持续改进”四段讲。提前参与指需求阶段确认边界、异常和验收标准；持续验证指开发过程中同步接口契约、单测、冒烟和回归；快速反馈指 CI 失败、核心链路异常和线上监控能及时暴露；持续改进指每轮迭代后复盘缺陷根因和自动化覆盖缺口。

敏捷测试不等于减少测试，也不等于只做自动化。它要求测试策略更有重点：高风险链路必须稳定回归，低风险变化可以轻量验证，新功能要结合探索测试发现未知问题。

## 标准回答框架

面试里回答敏捷测试，可以先说明它解决的是“短迭代下如何持续保证质量”的问题，而不是把原来的测试流程简单压缩。一个比较稳的回答结构是：第一，需求阶段左移，测试参与用户故事拆分、验收标准定义和风险识别；第二，开发阶段持续反馈，围绕接口契约、Mock、联调环境和冒烟用例提前发现偏差；第三，提测阶段重点验证，用自动化回归覆盖稳定链路，用探索性测试补充未知风险；第四，发布后闭环，通过监控、线上缺陷和复盘改进下一轮迭代。

这样讲能避免两个常见误区：一是把敏捷测试说成“开发完赶紧测”，二是把敏捷测试说成“全靠自动化”。敏捷测试真正强调的是协作、反馈和风险优先级。测试不再只是流程末端的检查者，而是从需求、设计、开发、发布到复盘都参与质量决策的人。

## 测试开发落地

测试开发在敏捷团队里的价值，通常体现在“让反馈更快、更稳定、更可重复”。例如在需求评审时，把验收标准沉淀成可执行的接口用例或 E2E 冒烟用例；在开发联调时，提供 Mock 服务、测试数据生成器和契约校验；在 CI 里维护 lint、类型检查、单测、接口回归和核心 UI 回归；在迭代复盘时，用缺陷逃逸率、自动化失败原因和线上监控数据反推流程短板。

如果项目节奏很快，测试策略要分层：每次提交跑快速门禁，保证代码和核心逻辑没有明显问题；每天或每个候选版本跑更完整的接口回归；发布前只保留高价值 UI 路径和关键业务链路。这样团队可以既快又稳，而不是在每个迭代末尾靠加班兜底。

## 面试追问

如果面试官追问“敏捷下测试时间被压缩怎么办”，可以回答：我会先按业务风险排序，而不是平均用力。核心链路、资金链路、权限链路必须进入自动化和门禁；低风险文案和样式调整可以轻量抽查；新功能的未知风险用探索性测试补充。之后把每次遗漏的问题复盘成用例、检查清单或流水线规则，让下一轮迭代不再重复踩坑。

如果追问“怎么衡量敏捷测试做得好不好”，不要只说发现多少 Bug。更好的指标包括：需求阶段发现的问题数、提测打回率、自动化回归耗时、核心用例通过率、缺陷逃逸率、线上回滚次数、从缺陷发现到修复的平均时间。指标不用于甩锅，而是帮助团队缩短反馈周期、减少重复问题。

## 快速记忆

敏捷测试可以记成一句话：需求阶段看风险，开发阶段给反馈，提测阶段抓重点，发布之后做闭环。面试时把“更早参与、更快反馈、更小批量验证、更持续改进”讲出来，就能和传统瀑布式测试区分开。

易错点是把敏捷测试说成“少写文档、少做测试”。更准确的表达是：文档更轻量但验收标准更明确，测试更聚焦但风险闭环更频繁，自动化更多但人工探索和团队沟通仍然不可替代。
