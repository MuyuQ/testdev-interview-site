## 业务流程

- 商品浏览与搜索。
- 购物车、下单、库存锁定与支付。
- 履约发货、售后退款与营销核销。

## 风险点

- 库存与订单状态不一致。
- 优惠叠加、价格计算和边界金额错误。
- 高并发活动下超卖、重复下单和回调延迟。

## 测试策略

- 核心链路做接口回归，前台关键页面做少量 UI 回归。
- 营销和价格相关规则做数据驱动组合测试。
- 下单支付链路增加日志与数据库双重校验。

## 可讲成果

- 核心下单回归从人工 3 小时缩到 25 分钟。
- 通过门禁阻断多次高风险价格配置上线。
- 补齐库存和支付异常链路后，线上重复单率下降。

## 面试讲法模板

讲电商项目时，不要从“我测了哪些页面”开始，而要先把业务闭环讲出来。可以这样说：

我参与的是电商交易链路测试，核心覆盖商品浏览、购物车、下单、库存锁定、支付回调、发货履约和售后退款。这个项目的测试重点不是页面按钮，而是订单状态、库存数量、优惠金额和支付流水之间的一致性，因为这些一旦出错会直接造成资损或用户投诉。

然后补一句你的职责：我主要负责核心交易链路的接口回归、营销规则的数据驱动测试、支付异常链路验证和活动前压测。这样面试官能快速听到你的业务理解、测试范围和技术抓手。

## 风险点拆解

### 价格和优惠风险

电商项目最敏感的是价格。满减、折扣、优惠券、会员价、活动价、运费、积分抵扣可能同时作用在一笔订单上。测试时要关注优惠叠加顺序、互斥规则、最低金额、退款后优惠返还和边界金额。面试里可以强调用数据驱动覆盖多种优惠组合，而不是手工点几单。

### 库存和订单状态风险

下单会涉及库存锁定、支付成功、支付超时、取消订单、退款退货等状态流转。风险在于库存扣减和订单状态不一致，例如支付失败但库存未释放，支付成功但订单仍是待支付，重复回调导致重复扣库存。测试要把订单状态机和库存变化放在一起验证。

### 支付回调风险

支付系统通常依赖第三方回调。回调可能延迟、重复、乱序或失败。测试要覆盖回调幂等、签名校验、金额校验、订单号校验、重复通知和补偿任务。资金链路不要只看前端支付成功页面，要看支付流水、订单状态和业务日志是否一致。

### 大促高并发风险

秒杀、促销、直播带货等场景下，请求量会突然升高。测试要关注超卖、重复下单、接口限流、排队、降级、缓存热点和数据库压力。面试时如果能讲出活动前容量评估、压测模型和监控指标，会比单纯说“做过性能测试”更有说服力。

## 测试策略拆解

- 核心链路：用接口自动化覆盖商品详情、购物车、下单、支付、发货、退款主流程，并把数据库状态和消息投递作为断言。
- 价格规则：把优惠类型、用户等级、商品类目、活动时间、库存状态作为参数做组合测试，重点覆盖边界金额和互斥规则。
- 状态机：为订单、支付、库存、售后分别画状态流转图，验证每个状态能否正确进入、退出和回滚。
- 异常链路：覆盖支付超时、重复回调、库存不足、优惠券失效、退款失败、消息消费失败和补偿任务。
- 大促保障：提前准备压测数据、热点商品、活动配置、监控面板和回滚方案，活动中观察下单成功率、库存扣减、支付成功率和错误率。

## 面试追问怎么接

如果面试官问“你发现过什么典型问题”，可以讲一个资损或一致性案例。例如：某次优惠券和满减叠加后，退款只退了实付金额但优惠券状态没有返还，导致用户二次下单无法使用券。定位时通过订单表、优惠券表、退款流水和接口日志确认问题，最后补了退款后券状态回滚的用例和自动化断言。

如果面试官问“自动化怎么做”，可以讲分层：交易核心链路用接口自动化保障稳定回归，营销规则用数据驱动生成组合用例，前端 UI 只覆盖关键路径，支付和库存相关断言下沉到数据库和日志。

## 容易答错什么

- 只讲页面功能，不讲交易链路。
- 只说做过支付测试，不讲回调幂等和资损风险。
- 只讲性能测试，不讲压测模型、容量指标和降级预案。
- 只说发现很多 Bug，不讲问题如何定位、如何沉淀成回归用例。
