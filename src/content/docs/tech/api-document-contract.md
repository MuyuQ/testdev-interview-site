## 说明

接口文档是团队协作的约定，契约测试用于验证接口实现是否符合文档定义，常用 OpenAPI 或 Pact。

## 面试怎么问

### 你们项目怎么维护接口文档？

使用 OpenAPI（Swagger）规范定义接口文档，包含：接口路径、请求方法、参数定义、响应结构、错误码。

文档管理方式：代码注解自动生成（保持同步）、独立文档仓库（集中管理）、文档平台（支持在线测试）。

关键是文档与代码同步更新，避免文档过时。

### 接口变更时怎么保证文档和实现一致？

建立变更管控流程：接口变更先更新文档评审、文档通过后同步开发、开发完成后做契约测试验证。契约测试在 CI 中执行：对比文档定义和实际响应，发现不一致自动报警。还要建立接口版本管理机制，兼容性变更升级小版本，破坏性变更升级大版本。

### 契约测试是什么，怎么用？

契约测试验证接口提供者和消费者是否符合约定的接口规范。

分为两种：提供者契约测试验证服务端是否符合文档定义。消费者契约测试验证客户端是否能正确处理服务端响应。

工具选择：OpenAPI Validator 用于 REST API，Pact 用于微服务契约测试。

契约测试是接口治理的重要手段。

## 项目里怎么用

- 用 OpenAPI 或 Swagger 定义接口规范，作为开发和测试的共同参考。
- 在 CI 中做契约测试，拦截不兼容变更，保证文档和实现一致。
- 文档变更时同步更新测试用例和[接口断言](/testdev-interview-site/glossary/api-assertion/)，避免测试覆盖遗漏。
- 使用文档生成测试用例骨架，提高测试编写效率。
- 建立接口变更通知机制，影响范围可控。

## 风险点拆解

### 文档和实现不一致

接口文档写的是一个契约，但真正调用时返回的是另一个结构，这是接口协作里最常见的问题。比如文档写 `amount` 是 number，实际返回 string；文档写字段必填，实际某些场景为空；文档写错误码是 `USER_NOT_FOUND`，实际返回通用错误码。这类问题会让调用方、自动化脚本和接口断言都变得不稳定。

### 破坏性变更未通知

接口新增字段通常风险可控，但删除字段、修改字段类型、改变枚举含义、调整错误码、修改分页规则都属于破坏性变更。测试要主动判断这次变更是否兼容，不能只看服务端自己跑通。

### 消费方覆盖不足

接口提供者测试通常只验证服务端返回正确，却不一定能发现客户端能否正确处理响应。消费者契约测试的价值就在这里：把调用方真实依赖的字段、错误码和边界场景固化下来，让提供方发版前就知道会不会影响调用方。

## 测试策略

- 文档基线：把 OpenAPI/Pact 文件纳入版本控制，变更必须经过评审和 CI 校验。
- Schema 校验：自动对比实际响应和文档 schema，检查字段存在性、类型、必填项和枚举。
- 兼容性校验：区分新增字段、可选字段、删除字段、类型变化、错误码变化，明确哪些是破坏性变更。
- 消费者契约：为关键调用方保留契约样例，验证提供方返回仍满足调用方依赖。
- 版本管理：重大接口调整通过 `/v2` 或版本头发布，保留旧版本兼容窗口。

## 面试追问

### 契约测试和普通接口测试有什么区别？

普通接口测试更关注当前接口能否按业务预期工作，契约测试更关注接口提供方和消费方的约定是否被破坏。它不一定覆盖所有业务组合，但会重点覆盖字段结构、类型、错误码、版本兼容和调用方依赖。

### 文档已经自动生成，还需要契约测试吗？

需要。自动生成只能说明代码里有文档注解，不代表真实运行时响应、异常场景和调用方依赖一定符合文档。契约测试把文档约定变成可执行校验，能在 CI 阶段阻断不兼容变更。

## 容易答错什么

### 文档和实现脱节，维护成本高

文档脱节的表现：接口参数变了文档没更新、错误码新增了没记录、响应结构与文档不符。

解决方案：代码注解自动生成文档、契约测试发现不一致、变更流程强制更新文档。

文档准确性直接影响前后端协作效率和测试效率。

### 不会讲契约测试和功能测试的区别

契约测试关注接口结构是否符合约定，功能测试关注业务逻辑是否正确。

契约测试验证：字段是否存在、类型是否正确、必填是否生效。

功能测试验证：业务规则、边界条件、异常处理。

两者互补，契约测试是功能测试的前提——接口结构都不对，功能测试无法进行。

### 忽略接口版本管理和兼容性策略

接口变更会影响所有调用方，要有版本管理策略：URL 路径版本化（/v1/api）、请求头版本化、向后兼容原则（新增字段可选、删除字段需要过渡期）。破坏性变更要提前通知、灰度发布、监控影响。版本管理是接口治理的核心能力。
