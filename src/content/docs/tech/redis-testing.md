## 说明

Redis 测试关注缓存读写、过期策略、数据一致性和缓存失效场景，是高并发业务测试的关键。

## 面试怎么问

### 你怎么验证缓存和数据库的一致性？

一致性验证要分场景：读取时验证缓存命中和数据库结果是否一致。更新时验证缓存是否同步更新或失效。删除时验证缓存是否清除。对于延时双删策略，要验证时间窗口内的一致性。还可以监控缓存命中率，异常低可能表示一致性问题。

### 缓存过期和失效场景怎么测？

测试场景包括：

缓存过期后重新加载。

缓存淘汰策略（LRU/LFU）。

缓存穿透（查询不存在的 key）。

缓存击穿（热点 key 过期瞬间大量请求）。

缓存雪崩（大量 key 同时过期）。

每种场景都要验证系统的行为是否符合预期，是否有兜底机制。

### Redis 在你项目里承担什么角色？

Redis 在项目中承担多种角色：

会话存储——验证过期和续期。

缓存层——验证一致性。

分布式锁——验证竞争和超时。

消息队列——验证消费确认。

计数器——验证原子性和准确性。

不同角色的测试重点不同。

## 项目里怎么用

- 在接口测试中验证缓存读写结果和过期时间，确保缓存逻辑正确。
- 覆盖缓存穿透、缓存雪崩和缓存击穿场景，验证兜底机制。
- 结合数据库变更验证缓存更新和失效逻辑，确保一致性。
- 测试分布式锁的竞争、超时和释放场景。
- 验证 Redis 高可用场景（主从切换、哨兵模式）。

## 测试策略拆解

### 读缓存链路

读链路要验证命中缓存、未命中缓存、缓存过期后回源数据库、回源成功后重新写缓存。测试时可以先准备一条确定数据，清空 key 后请求接口，确认第一次未命中会查询数据库并写入缓存；第二次请求应命中缓存；修改数据库后如果业务要求立即一致，要验证缓存是否被更新或删除。

### 写缓存链路

写链路要先确认系统采用什么策略：先更新数据库再删缓存、先删缓存再更新数据库、延时双删、异步消息失效，还是写穿缓存。不同策略的测试重点不同。面试里可以主动说明“我不会只测写入成功，而是会测写入后读到的数据是否符合一致性约束”。

### 异常链路

Redis 异常时要验证系统是否有兜底。例如缓存连接超时是否会降级到数据库，缓存写入失败是否影响主交易，缓存删除失败是否有补偿任务，热点 key 被删除是否会触发数据库压力保护。高并发业务不能把 Redis 当成绝对可靠组件。

### 并发链路

并发场景要关注锁、计数器、库存、限流和幂等 key。测试时可以构造多线程或压测请求，验证 Redis 原子操作是否保护了业务状态，避免出现库存扣减错误、重复提交、限流不生效或锁被误释放。

## 面试表达模板

如果面试官问“你怎么测 Redis”，可以这样组织：

第一，我会先确认 Redis 在系统中的角色，是缓存、分布式锁、限流计数器、会话存储还是消息队列。

第二，我会按角色拆风险。缓存关注一致性、过期、穿透、击穿、雪崩；锁关注竞争、超时、续期和释放；计数器关注原子性和边界；会话关注过期和续期。

第三，我会设计正常、异常、并发和高可用场景。正常场景验证读写和过期，异常场景验证 Redis 超时和故障兜底，并发场景验证原子性，高可用场景验证主从切换和数据恢复。

第四，我会补监控和定位能力，包括命中率、慢查询、内存使用、key 数量、过期 key、连接数、错误率和数据库回源量。缓存问题不只靠功能用例发现，很多要靠指标趋势判断。

## 容易答错什么

### 只说会用 Redis，不会讲缓存测试场景

Redis 测试不是简单的读写验证，要覆盖：缓存命中/未命中场景、缓存更新策略、过期和淘汰行为、缓存异常（连接失败、超时）的兜底、并发访问下的正确性。

每种场景都有不同的测试设计，要展示完整的测试思路。

### 忽略缓存与数据库的一致性验证

缓存一致性问题可能导致严重的数据错误：用户看到过期数据、操作结果丢失。测试要验证：更新操作后缓存是否同步、删除操作后缓存是否清除、异常情况下是否有补偿机制。这是缓存测试的核心价值。

### 不会讲缓存失效对业务的影响

缓存失效的影响取决于业务场景：热点数据缓存失效可能导致数据库压力骤增。会话缓存失效导致用户登出。分布式锁失效导致并发问题。要分析业务对缓存的依赖程度，设计对应的测试场景和容灾验证。
