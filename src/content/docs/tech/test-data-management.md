## 说明

测试数据管理关注数据准备方式、环境隔离、数据清理和数据复用策略，是自动化稳定性的基础。涉及 [Fixture](/testdev-interview-site/glossary/fixture/) 管理和[测试环境管理](/testdev-interview-site/glossary/test-environment-management/)。

## 面试怎么问

### 你们项目的测试数据怎么准备和管理？

数据准备方式：通过 API 创建（走业务逻辑，数据一致性好）、数据库插入（快速，适合批量数据）、配置文件定义（静态数据）。

管理策略：按场景分类（基础数据、业务数据、边界数据）、统一数据工厂类生成、数据模板支持参数化覆盖。

关键是数据准备要可重复、可追溯。

### 如何保证测试数据不影响其他环境？

环境隔离策略：使用独立的测试数据库、通过命名空间或标签区分测试数据、测试账号与生产账号隔离。数据隔离：每个测试使用唯一标识（如时间戳、UUID）、测试数据有明确的创建者标记、测试后清理或归档。还要防止测试数据泄露到监控系统或报表。

### 数据复用和数据隔离怎么取舍？

核心原则：对测试结果有影响的数据要隔离，对测试结果无影响的共享数据可以复用。

隔离的数据：订单、支付、用户状态等业务数据。

复用的数据：配置数据、字典数据、测试账号池。

复用提高效率，隔离保证稳定，要根据测试场景找到平衡点。

## 项目里怎么用

- 通过 API、数据库或配置文件准备测试数据，确保数据一致性。
- 用独立数据库或命名空间隔离测试环境，避免数据污染。
- 每个测试后自动清理或标记回收数据，保持环境干净。
- 建立数据工厂类，支持链式调用生成复杂测试数据。
- 数据模板版本化管理，与测试代码同步更新。

## 风险点拆解

### 数据污染导致用例不稳定

自动化测试最怕“昨天能过、今天不能过”，很多时候根因是数据被其他用例改掉了。比如一个订单被用例 A 取消后，用例 B 还拿它做支付；一个测试账号被锁定后，所有登录用例都失败。对会被修改的数据要做到用例级隔离。

### 测试数据不可追溯

如果测试数据没有创建者、场景标识和生命周期，出了问题很难判断是谁创建的、为什么存在、能不能删除。建议为自动化数据加统一前缀、标签、创建时间和场景编号。

### 数据准备绕过业务规则

直接插库很快，但可能绕过业务校验、审计、缓存和消息，导致数据状态不真实。关键链路建议优先通过 API 创建数据，只有大批量静态数据或边界数据才考虑直接插库，并补充一致性校验。

## 面试追问

### 你们怎么清理测试数据？

我会按数据类型选择策略：用例级临时数据测试后立即清理；可复用基础数据长期保留但只读；复杂业务数据用标签和过期时间定期归档或删除。清理动作要谨慎，避免误删共享数据。

### 随机数据和固定数据怎么取舍？

固定数据便于复现和断言，随机数据能避免唯一性冲突。通常核心断言用固定可控数据，唯一字段加随机后缀；大量边界组合通过数据模板生成，并把失败时的输入数据记录到报告里方便复现。

## 容易答错什么

### 数据散落各处，没有统一管理策略

数据管理混乱的表现：数据定义散落在各测试文件、相同数据多处重复定义、数据变更难以追踪。

解决方案：建立数据目录集中管理、使用数据工厂统一生成、数据模板支持继承和覆盖。

统一管理提高维护效率，也便于数据变更时的批量更新。

### 数据残留导致后续测试干扰或失败

数据残留是自动化测试不稳定的常见原因。解决策略：测试后立即清理（推荐）、测试前重置环境（保证初始状态）、定期批量清理（兜底）。清理要彻底：不仅清理数据库，还要清理缓存、消息队列、文件等。建立数据清理检查机制，确保清理生效。

### 不会讲数据版本与环境版本的对齐

数据版本要与代码版本、环境版本对齐：数据结构变更时同步更新数据模板、环境升级时检查数据兼容性、多环境并行时数据要独立。版本对齐机制：数据变更记录 CHANGELOG、数据版本号与代码版本对应、环境配置与数据配置绑定。
